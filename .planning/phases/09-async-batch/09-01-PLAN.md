Modify `extract_with_retry()` in `policy_extractor/extraction/client.py` to add an inner rate limit retry loop wrapping the `call_extraction_api()` call.

CRITICAL PLACEMENT: The rate limit retry MUST wrap `call_extraction_api()` INSIDE the existing `for attempt in range(attempts)` loop, BEFORE the `except Exception` clause at line 126. If placed outside `extract_with_retry()`, the broad `except Exception` at line 126 swallows the RateLimitError before it reaches the outer wrapper. See RESEARCH.md Pitfall 1.

Add these imports at the top of client.py:
```python
import random
import time
```

Add module-level constants:
```python
_RATE_LIMIT_MAX_RETRIES = 3
_RATE_LIMIT_BACKOFF = [2, 4, 8]  # seconds per retry attempt
```

Modify `extract_with_retry()` signature to add `max_rate_limit_retries: int = 3` parameter. The return type stays `tuple[PolicyExtraction, dict, anthropic.types.Usage] | None` but the function now also returns the rate limit retry count. Change to return a 4-tuple: `(policy, raw_response, usage, rate_limit_retries_used)` on success, or `None` on failure.

Inside the `for attempt in range(attempts)` loop, replace the direct `call_extraction_api()` call with a while-loop:

```python
rl_retries = 0
for rl_attempt in range(max_rate_limit_retries + 1):
    try:
        message = call_extraction_api(client, current_text, model)
        break  # API call succeeded
    except (
        anthropic.RateLimitError,
        anthropic.InternalServerError,
        anthropic.APIConnectionError,
    ) as exc:
        if rl_attempt >= max_rate_limit_retries:
            raise  # exhausted retries, let outer except handle it
        wait = _RATE_LIMIT_BACKOFF[rl_attempt] + random.uniform(0, 1)
        rl_retries += 1
        logger.warning(
            "[RETRY] Rate limit/transient error (attempt %d/%d, waiting %.1fs): %s",
            rl_attempt + 2, max_rate_limit_retries + 1, wait, exc,
        )
        time.sleep(wait)
```

After the rate limit loop, the existing `parse_and_validate()` call remains. On success, return the 4-tuple: `return (policy, raw_response, message.usage, rl_retries)`.

The existing `except ValidationError` and `except Exception` handlers remain unchanged, but when returning None in those handlers, they effectively discard the retry count (acceptable -- failures don't contribute to retry summary).

ALSO update the caller in `policy_extractor/extraction/__init__.py` to handle the new 4-tuple. Change line 58 from:
```python

```
to:
```python
policy, raw_response, usage = outcome[0], outcome[1], outcome[2]
# outcome[3] is rate_limit_retries_used -- not needed in extract_policy()
```
OR more cleanly, unpack with `*_`:
```python
policy, raw_response, usage, *_ = outcome
```

This keeps `extract_policy()` backward compatible -- it still returns `(PolicyExtraction | None, Usage | None)`.
